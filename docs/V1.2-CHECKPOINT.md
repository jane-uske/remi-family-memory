# v1.2 Checkpoint: VLM-Assisted Draft Enrichment

**Released:** 2026-05-19
**Scope:** OCR-first + VLM-assisted pending draft enrichment, parent confirmation required

---

## What v1.2 adds

Local VLM (Vision Language Model) can now analyze image attachments in pending drafts and generate structured metadata:

- `inferredTitle` — short descriptive title
- `inferredDate` — ISO date if visible in the image
- `inferredType` — event type (pregnancy_checkup, milestone, etc.)
- `extractedFacts` — observable text/values from the image (e.g. "CRL: 7.2cm")
- `uncertainFields` — fields the VLM is not confident about

---

## What VLM writes

| Field | Written to | Scope |
|-------|-----------|-------|
| `inferredTitle` | pending draft JSON | Only fills null fields, never overwrites |
| `inferredDate` | pending draft JSON | Only fills null fields |
| `inferredType` | pending draft JSON | Only fills null fields, validated against allowed types |
| `extractedFacts` | pending draft JSON | Array of directly-visible text/values |
| `extractionMetadata` | pending draft JSON | Model name, timestamp, attachment ID |
| `uncertainFields` | pending draft JSON | Merged with existing, no duplicates |

## What VLM does NOT write

| Target | Written? | Reason |
|--------|----------|--------|
| `events.json` (confirmed events) | **Never** | Only `confirmDraft()` + `scanInbox()` can write here |
| `memories.json` (AI-queryable memory) | **Never** | Only `buildMemories()` from confirmed events |
| `context/remi-context.json` | **Never** | Only `generateContext()` from AI-safe (confirmed) events |
| `draft.status` | **Never changes** | Remains `pending` after enrichment |
| `draft.captureStatus` | **Never changes** | Remains `pending_parent_review` |

## Why VLM cannot bypass parent confirmation

```
Image uploaded → intake creates pending draft
                       ↓
              VLM enriches draft (optional, may fail)
                       ↓
              Parent reviews draft in Remi
                       ↓ (explicit "确认")
              confirmDraft() writes inbox note with confirmedByParent: true
                       ↓
              npm run sync → scanInbox() → events.json
                       ↓
              buildMemories() → memories.json (AI-queryable)
```

VLM output is **advisory only**. It cannot:
1. Change draft status
2. Write to confirmed storage
3. Influence Remi's answers to family questions (query path only reads confirmed memory)

## VLM failure degradation

| Failure | Behavior |
|---------|----------|
| `VLM_MODEL` not set | Returns `no_vlm_config`, draft unchanged |
| VLM service unreachable | Returns `vlm_failed`, draft unchanged |
| VLM timeout (default 120s) | Returns `vlm_failed`, draft unchanged |
| VLM returns unparseable output | Returns `vlm_failed`, draft unchanged |
| Image >10MB | Skipped, draft unchanged |
| No image attachments (PDF-only) | Returns `no_image_attachments`, draft unchanged |

In all failure cases, the draft remains fully usable: parents can still supplement summary manually and confirm.

## Medical safety guard

VLM system prompt bans conclusive medical language. `validateVlmOutput()` checks for patterns like:
- "很健康" / "一切正常" / "没问题" / "无异常"
- "不用担心" / "可以放心" / "诊断为" / "确诊"

Any detected pattern is logged as a validation warning. VLM `needsParentReview` is force-set to `true` regardless of model output.

## API surface

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/ai/drafts/:id/enrich` | POST | Enrich single pending draft |
| `/api/ai/drafts/enrich-all` | POST | Enrich all pending drafts |

CLI: `npm run enrich-draft <id>` / `npm run enrich-drafts`
